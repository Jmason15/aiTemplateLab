# aiTemplateLab — Claude Context

## What This App Does

A single-page browser app for creating, managing, and exporting structured AI prompt templates. Templates follow a 7-section schema (Objective, Actor, Context, Inputs, Constraints, Outputs, Success Criteria) and are persisted to `localStorage`. The build step compiles everything into one self-contained HTML file with no external dependencies.

---

## Build System

```bash
node build.js          # produces dist/aiTemplateLab.html
npm run build          # same, via package.json script
```

`build.js` performs four steps:
1. **Generate prompt data** — reads `main/config/workspaces.json`, loads each referenced `main/Prompts/*.json` file, and builds the `preloadedPrompts` / `preloadedWorkspaces` / `preloadedConfig` JS string in memory.
2. **Resolve HTML partials** — replaces `<!-- #include "path/to/partial.html" -->` directives with file contents.
3. **Inline CSS** — replaces all `<link rel="stylesheet">` tags with a single `<style>` block containing all CSS files concatenated in order.
4. **Inline JS** — replaces each `<script src="js/...">` tag with an inline `<script>` block. `preloadedPrompts.js` is special: it is generated in memory (step 1) and never exists on disk.

**The dist file is the only deployable artifact.** After any source change, rebuild and commit both the source and `dist/aiTemplateLab.html` together.

## Adding a Prompt Template

1. Create a JSON file in `main/Prompts/` following the data model below. The file can be a plain object `{}` or an array `[{}, {}]`.
2. Add the filename to `main/config/workspaces.json` under the appropriate workspace.
3. Run `node build.js`.

**`workspaces.json` shape:**
```json
{
    "workspaces": {
        "Default": ["promptBuilder.json"],
        "Jira": ["jiraStory.json"]
    },
    "defaultWorkspace": "Jira",
    "defaultTemplate": "Jira Story and Acceptance Criteria Generator"
}
```

`defaultWorkspace` controls which group loads on first open. `defaultTemplate` controls which template within that group opens. You do not need to touch any JS files to add a new prompt.

## Data Model

Every template has this shape:

```json
{
  "id": "uuid-string",
  "name": "Template name",
  "description": "Short description",
  "objective": "What this prompt accomplishes",
  "actor": "AI persona/role",
  "context": "Background context for the AI",
  "inputs": [
    { "name": "fieldName", "description": "what to provide" }
  ],
  "constraints": ["rule 1", "rule 2"],
  "outputs": [
    { "name": "outputName", "type": "string", "description": "what comes back" }
  ],
  "success": ["criterion 1", "criterion 2"]
}
```

**localStorage keys:**
- `templateGroups` — `{ [groupName]: prompt[] }`
- `templateGroupHistory` — `{ [groupName]: inputHistory[] }`
- `currentTemplateGroup` — active group name string

## State Management

State is global — no framework. The authoritative variables are declared as `let` in `state.js` and are accessible to all later-loaded scripts.

| Variable | Declared in | Purpose |
|---|---|---|
| `prompts` | state.js | Flat working array of templates in the active group |
| `currentPromptId` | state.js | ID of the currently selected template |
| `currentTemplateGroup` | state.js | Active workspace/group name |
| `environment` | state.js | `{ templateGroups: {}, history: {} }` — full authoritative store |

**Dual-write pattern:** `prompts[]` (flat working array) and `environment.templateGroups[currentTemplateGroup][]` (authoritative source) must always be kept in sync. Any mutation should update both and call `savePromptsToLocalStorage()`.

**`window.*` sync:** `editPrompt.js` reads state via `window.*` (e.g. `window.currentPromptId`) because it was written independently of `state.js`. `syncWindowState()` in `state.js` keeps `window.*` mirrors in sync. Call it after any state mutation.

---

## Key Patterns

**`normalizePrompt()`** — guards against legacy data where `constraints` or `success` entries were stored as objects instead of strings. Always call this when loading prompts from storage or files: `prompts.map(normalizePrompt)`.

**Debounced auto-save** — in `editPrompt.js`, every field input event fires `debouncedSaveCurrentPrompt` (200ms delay). `saveCurrentPrompt()` uses counter-based iteration (`for i = 1..inputCounter`) rather than `querySelectorAll` so it correctly skips elements that were deleted and removed from the DOM.

**`window.regenerateOutput`** — defined in `promptCrud.js`. Called by `editPrompt.js` on every field change. Guards against running while the edit screen is not active.

**`resetToPreloaded()`** — iterates `window.preloadedWorkspaces` dynamically (not hardcoded names), so adding new workspaces to `workspaces.json` requires no code change.

## Adding a New Template Section

To add a new section to the prompt schema (e.g., "Examples"):

1. Add HTML for the section in `main/index.html` (edit form area) and any relevant partial.
2. Add an `addExample()` function in `editPrompt.js` following the `addTextareaField()` pattern.
3. Update `saveCurrentPrompt()` in `editPrompt.js` to read the new fields.
4. Update `viewPrompt()` in `promptCrud.js` to display the new fields.
5. Update `generateViewPrompt()` in `promptOutput.js` to include them in JSON output.
6. Rebuild: `node build.js`.

---

## CSS File Order

CSS files must be added to `CSS_FILES_ORDERED` in `build.js` **and** to `<link>` tags in `main/index.html` in the same order. `variables.css` must always be first (defines all custom properties used by other files).

## Gotchas

- `preloadedPrompts.js` is generated in memory by `build.js` — never create or commit this file manually.
- There is no dev server and no test suite. Open `main/index.html` directly in a browser for local dev; rebuild to produce the dist file.
- `localStorage` is per-origin. Testing `main/index.html` via `file://` and `dist/aiTemplateLab.html` share the same origin, so they share localStorage state.
- The app has no undo. Deleting a template is permanent.
- No linter or formatter is configured. Be consistent with the surrounding code style.
