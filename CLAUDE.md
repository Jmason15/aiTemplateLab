# aiTemplateLab — Claude Context

## What This App Does

A single-page browser app for creating, managing, and exporting structured AI prompt templates. Templates follow a 7-section schema (Objective, Actor, Context, Inputs, Constraints, Outputs, Success Criteria) and are persisted to `localStorage`. The build step compiles everything into one self-contained HTML file with no external dependencies.

The app always opens on the home screen with no template selected. The sidebar shows all template groups as collapsible sections with a search bar at the top. A Windows-style menu bar (File | Groups | Templates | Help) replaces the old dropdown/kebab pattern.

---

## Build System

```bash
node build.js          # produces dist/aiTemplateLab.html and docs/index.html
npm run build          # same, via package.json script
```

`build.js` performs four steps:
1. **Generate prompt data** — reads `main/config/workspaces.json`, loads each referenced `main/Prompts/*.json` file, and builds the `preloadedPrompts` / `preloadedWorkspaces` / `preloadedConfig` JS string in memory.
2. **Resolve HTML partials** — replaces `<!-- #include "path/to/partial.html" -->` directives with file contents.
3. **Inline CSS** — replaces all `<link rel="stylesheet">` tags with a single `<style>` block containing all CSS files concatenated in order.
4. **Inline JS** — replaces each `<script src="js/...">` tag with an inline `<script>` block. `preloadedPrompts.js` is special: it is generated in memory (step 1) and never exists on disk.

The build also:
- Base64-encodes `images/aiTemplateLab.png` and injects it as the app bar logo (`__LOGO_SRC__` placeholder).
- Base64-encodes `images/favicon.png` (small, <100KB) and injects it as the browser tab icon (`__FAVICON_SRC__` placeholder). Falls back to the GitHub Pages URL if the file is missing.
- Copies both images to `docs/` so the OG image meta tag resolves correctly on GitHub Pages.

**The dist file is the only deployable artifact.** After any source change, rebuild and commit both the source and `dist/aiTemplateLab.html` together.

## CI/CD — GitHub Actions

Defined in `.github/workflows/release.yml`. Triggers on any tag matching `release_*`.

**What it does:**
1. Runs `node build.js`
2. Deploys `docs/` to GitHub Pages
3. Creates a GitHub Release with `dist/aiTemplateLab.html` attached

**To release a new version:**
```bash
git tag release_YYYY_MM_DD
git push origin release_YYYY_MM_DD
```

**One-time setup required:** In the repo's GitHub Settings → Pages → Build and deployment, set the source to **GitHub Actions** (not "Deploy from a branch").

## Adding a Prompt Template

1. Create a JSON file in `main/Prompts/` following the data model below. The file can be a plain object `{}` or an array `[{}, {}]`.
2. Add the filename to `main/config/workspaces.json` under the appropriate workspace.
3. Run `node build.js`.

**`workspaces.json` shape:**
```json
{
    "workspaces": {
        "Template Lab": ["templateBuilder.json", "groupGenerator.json", "workflowGenerator.json"],
        "Example Templates": ["jiraStory.json", "codeCleaner.json"]
    }
}
```

No `defaultWorkspace` or `defaultTemplate` — the app always opens on the home screen. You do not need to touch any JS files to add a new prompt.

## Data Model

Every template has this shape:

```json
{
  "id": "readable-slug",
  "name": "Template name",
  "description": "Short description shown in the sidebar info bar",
  "objective": "What this prompt accomplishes",
  "actor": "AI persona/role",
  "context": "Background context for the AI",
  "example": "Plain-English walkthrough shown in the 'See an Example' section (optional)",
  "inputs": [
    { "name": "fieldName", "description": "what to provide", "placeholder": "example value shown in the textarea" }
  ],
  "constraints": ["rule 1", "rule 2"],
  "outputs": [
    { "name": "outputName", "type": "string", "description": "what comes back" }
  ],
  "success": ["criterion 1", "criterion 2"]
}
```

**localStorage keys:**
- `templateGroups` — `{ [groupName]: prompt[] }`
- `templateGroupHistory` — `{ [groupName]: inputHistory[] }`
- `currentTemplateGroup` — active group name string

## State Management

State is global — no framework. The authoritative variables are declared as `let` in `state.js` and are accessible to all later-loaded scripts.

| Variable | Declared in | Purpose |
|---|---|---|
| `prompts` | state.js | Flat working array of templates in the active group |
| `currentPromptId` | state.js | ID of the currently selected template |
| `currentTemplateGroup` | state.js | Active workspace/group name |
| `environment` | state.js | `{ templateGroups: {}, history: {} }` — full authoritative store |

**Dual-write pattern:** `prompts[]` (flat working array) and `environment.templateGroups[currentTemplateGroup][]` (authoritative source) must always be kept in sync. Any mutation should update both and call `savePromptsToLocalStorage()`.

**`window.*` sync:** `editPrompt.js` reads state via `window.*` (e.g. `window.currentPromptId`) because it was written independently of `state.js`. `syncWindowState()` in `state.js` keeps `window.*` mirrors in sync. Call it after any state mutation.

## Key Patterns

**`normalizePrompt()`** — guards against legacy data where `constraints` or `success` entries were stored as objects instead of strings. Always call this when loading prompts from storage or files: `prompts.map(normalizePrompt)`.

**Debounced auto-save** — in `editPrompt.js`, every field input event fires `debouncedSaveCurrentPrompt` (200ms delay). `saveCurrentPrompt()` uses counter-based iteration (`for i = 1..inputCounter`) rather than `querySelectorAll` so it correctly skips elements that were deleted and removed from the DOM.

**`window.regenerateOutput`** — defined in `promptCrud.js`. Called by `editPrompt.js` on every field change. Guards against running while the edit screen is not active.

**`resetToPreloaded()`** — iterates `window.preloadedWorkspaces` dynamically (not hardcoded names), so adding new workspaces to `workspaces.json` requires no code change.

**`window.showHelpModal(key)`** — exposed by `workspaceManager.js`. Accepts a key matching the `helpContent` map (e.g. `'help-what-is-template'`). Used by home screen chips and the Help menu to open the help modal without duplicating content.

**Sidebar search** — `renderPromptsList()` reads `#sidebar-search` value on every call. When a query is present it filters all groups and shows only matching tiles with all sections expanded. When empty it renders the normal grouped view.

**Edit screen accordions** — `setupEditAccordions()` in `app.js` wires header click → toggle `.open` on the parent `.edit-accordion`. `expandEditAccordion(bodyId)` (also in `app.js`, exposed as `window.expandEditAccordion`) forces a section open — called by the "+ Add" buttons so adding an item always reveals the section. `setupEditCounts()` uses `MutationObserver` on each container to keep the count badge in each accordion header in sync.

**Quick export / share** — `setupQuickExportModal()` in `importExport.js` wires the quick export modal. `shareOrDownload(data, filename, title)` attempts `navigator.share()` with a File object; falls back to `downloadJson()` if the Web Share API is unavailable or the user cancels. The builder screen's "Adding to" group select defaults to "Template Lab" if that group exists (`refreshBuilderGroupSelect()` in `importExport.js`).

**Import modal** — `openNewPromptModal()` opens a file-drop modal (no "From AI" tab). `processImportFile(file)` auto-detects the file type: `{ name, templates }` shape → group import handled inline; anything else → delegates to `showImportModal()` for checkbox-based template selection.

## UI Structure

**App bar** — fixed top bar with the app name (home button) and a Windows-style menu bar: File | Groups | Templates | Help. Each top-level button toggles its dropdown; hovering switches between open dropdowns.
- **File**: Save Workspace, Load Workspace, Reset, Live Site, Download Latest
- **Groups**: New, Export, Delete (no Load — use the sidebar Import button instead)
- **Templates**: New Blank, Export (no import items — use the sidebar Import button)
- **Help**: help topic buttons that open the help modal

**Sidebar** — fixed left panel containing:
- Search bar at top
- Green **Build a New Template** tile (primary creation path — opens the builder screen)
- Collapsible group sections; active group open by default. If the active group has no visible templates, the first group that does is auto-expanded.
- **Import** button at the bottom — opens a file-drop modal (auto-detects group vs. template file)
- **Export / Share** button — opens the quick export modal (This Template / This Group / Everything, each with Download + Share via Web Share API)

**Template use view** (`#template-view`) — replaces the old tab-based chrome. Shown when a template is selected. Four numbered steps:
1. Review template (Show Details collapsible)
2. Fill in inputs (with History dropdown and Clear button)
3. Copy Prompt button
4. Paste into AI instruction + View Prompt JSON collapsible

**Edit screen** (`#edit-screen`) — shown when editing a template. No tab bar, no info header. Structure:
- Back button
- Template name input (always visible, editable)
- Description textarea (always visible, editable)
- **Template Details** accordion (collapsed by default) — objective, actor, context, example
- **Inputs**, **Constraints**, **Outputs**, **Success Criteria** accordions (collapsed by default, show item count badge)
- Sticky footer with Cancel and Delete buttons (no Save — changes auto-save on every keystroke via 200ms debounce)

**Home screen** — always shown on startup and when clicking the app name. Contains a hero section, 3-step how-to, and four collapsible accordions with help chip links.

## Adding a New Template Section

To add a new section to the prompt schema (e.g., "Examples"):

1. Add HTML for the section in `main/index.html` (edit form area) and any relevant partial.
2. Add an `addExample()` function in `editPrompt.js` following the `addTextareaField()` pattern.
3. Update `saveCurrentPrompt()` in `editPrompt.js` to read the new fields.
4. Update `viewPrompt()` in `promptCrud.js` to display the new fields.
5. Update `generateViewPrompt()` in `promptOutput.js` to include them in JSON output.
6. Rebuild: `node build.js`.

## CSS File Order

CSS files must be added to `CSS_FILES_ORDERED` in `build.js` **and** to `<link>` tags in `main/index.html` in the same order. `variables.css` must always be first (defines all custom properties used by other files).

## Gotchas

- `preloadedPrompts.js` is generated in memory by `build.js` — never create or commit this file manually.
- There is no dev server and no test suite. Open `main/index.html` directly in a browser for local dev; rebuild to produce the dist file.
- `localStorage` is per-origin. Testing `main/index.html` via `file://` and `dist/aiTemplateLab.html` share the same origin, so they share localStorage state.
- The app has no undo. Deleting a template is permanent.
- No linter or formatter is configured. Be consistent with the surrounding code style.
- `base.css` applies `details { background: #f0f0f0; padding: 8px 12px }` globally. Override per-component (e.g. `.group-section` in sidebar.css uses `background: transparent; padding: 0`).
- The sidebar's generic `#sidebar button` rule overrides btn classes — use ID-specific rules with `!important` when a sidebar button needs a distinct colour (e.g. `#builder-nav-tile` for the green Build tile, `#import-btn` / `#export-btn` for the footer buttons).
- Never add inline `style="display:..."` to elements whose visibility is controlled by CSS media queries — inline styles have higher specificity and will silently break responsive behaviour. Manage show/hide via CSS classes only.
- The favicon must stay under ~100KB. Base64-encoding a large image as a `data:` URI exceeds browser limits and the icon will silently not appear.
